## 1. Keep（良かったこと・継続すべきこと）

### K-1: 依存関係を意識した計画的な開発順序

インフラ基盤（DAY1）→ バックエンド API（DAY2〜3）→ OP バックエンド（DAY3〜4）→ フロントエンド（DAY4〜5）と、依存関係を正しく反映したスケジュールを組み、その通りに実行できた。後続タスクのブロッキングが一切発生せず、スムーズな開発フローを維持できた。（伊藤・田中）

### K-2: 初回スプリントで全 PBI 完了・消化率 100%

初回であるにも関わらず 46SP を 100% 消化し、11 PBI 全てを完了できた。計画段階のキャパシティ見積もり（70h に対し 57h、バッファ率 19%）が妥当であったことが実証された。（高橋）

### K-3: バックエンドとフロントエンドの並行開発体制

伊藤がバックエンドを主導、田中がインフラ・フロントエンドを主導する役割分担が明確に機能した。PBI-011 や PBI-012 では両名がタスクを分担して並行作業し、効率的に進められた。（鈴木）

### K-4: テスト品質の確保

87 件のテストを全て作成・パスさせ、完成の定義のテスト基準を着実に満たした。特にバックエンド API では正常系・異常系を網羅し、フロントエンドでも vitest + Testing Library を導入して 21 件のコンポーネントテストを整備した。（伊藤）

### K-5: 構造化ログモジュールの早期完成と全 API への統一適用

PBI-006 を DAY1 で先行着手し DAY2 で完成させたことで、全 Lambda 関数に統一的な構造化ログ出力（JSON 形式、必須フィールド網羅）を適用できた。オブザーバビリティの基盤としてプロダクトゴールに直結する成果。（伊藤・田中）

### K-6: 軽微な問題の自律的な解消

boto3 / AWS_DEFAULT_REGION の未設定問題（DAY2）、EndpointsTable の HTML 構造問題（DAY5）など、発生した問題をチーム内で即座に解消し、障害物ログへの正式記載が不要なレベルで対処できた。自己管理型チームとして良い動きだった。（高橋）

### K-7: PBI-013（時間範囲セレクタ）の実質的な前倒し完了

PBI-014 の実装時に PBI-013 の機能を自然に内包して完成させた。バックログ上の PBI 境界にとらわれず、価値ある成果を効率的に届ける判断ができた。（鈴木）

## 2. Problem（問題だったこと・改善が必要なこと）

### P-1: Must 要件の未実装（OP-F-14, OP-F-15, Logs 画面）

顧客（佐藤）が条件付き受入とした Must 要件がスプリント内で完了しなかった。Endpoint Detail 画面のエラー数推移グラフ（OP-F-14）、ステータスコード内訳（OP-F-15）、および Logs 画面（OP-S-04）のフロントエンド実装が未完了。これらはプロダクトゴールの「問題の迅速な特定」に直結する機能であり、今スプリントではバックログの選択範囲外だったが、顧客期待との認識ギャップが生じた。（鈴木・高橋）

### P-2: Terraform CLI 未インストールによる terraform plan 未検証

完成の定義の「6. インフラストラクチャ」に「`terraform plan` が正常に完了すること」と明記されているが、開発環境に Terraform CLI がインストールされていなかったため、全スプリントを通じて未検証のまま完了した。DoD を厳密に適用するならば、この基準は満たされていない。（高橋）

### P-3: フロントエンド開発が DAY4〜5 に集中

フロントエンド実装が DAY4〜5 の 2 日間に集中し、テスト作成は最終日に行われた。余裕が少なく、もし問題が発生した場合のバッファが限られていた。結果的にEndpointsTable の HTML 構造問題が DAY5 で発覚し修正を要した。（田中）

### P-4: 障害物ログの未活用

軽微とはいえ、Terraform CLI 未インストール・boto3 環境問題・HTML 構造問題の 3 件が発生したが、いずれも正式な障害物ログに記載されなかった。問題の追跡・可視化という観点では改善の余地がある。透明性の柱を強化するためにも、記録の閾値を明確にすべき。（高橋）

## 3. Try（次スプリントで試したいこと）

### T-1: 顧客 Must 要件の優先的な着手

Sprint 002 では OP-F-14, OP-F-15, Logs 画面（OP-S-04）の Must 要件を最優先で着手する。顧客期待との認識ギャップを解消し、条件付き受入の条件を確実に満たす。（鈴木・チーム全員）

### T-2: 開発環境のセットアップ手順書作成と terraform plan 実行

Terraform CLI など必要なツールの一覧とインストール手順を整理し、`terraform plan` による検証を実際に実施する。CI/CD パイプラインでの自動検証も検討する。（田中）

### T-3: フロントエンド・バックエンド開発の時間分散

フロントエンド実装を特定の日に集中させず、バックエンド API の完成に合わせて段階的に着手する。DAY3 頃からフロントエンド開発を開始できるよう計画を工夫する。（田中・伊藤）

### T-4: 障害物の記録基準の明確化

「当日中に解消できない問題」または「他メンバーのタスクに影響を与える問題」を障害物ログに記録する基準とし、軽微な問題もデイリースクラム記録には必ず残す運用を徹底する。（高橋）

## 5. 完成の定義（DoD）の見直し

### 議論内容

| 項目 | 現行 | 提案 | 理由 |
|------|------|------|------|
| 6. インフラストラクチャ | `terraform plan` が正常に完了すること | **`terraform plan` が正常に完了すること（Terraform CLI が利用可能な環境で検証済み）** | Terraform CLI 未インストール問題を受け、検証環境の存在を前提条件として明記する。環境が利用不可の場合は障害物として記録し、対応する |

### チーム合意

- **鈴木（PO）**: DoD の基準自体は変更不要だが、検証できなかった場合の対応フローを明確にすべき。賛成。
- **伊藤（Dev）**: Terraform CLI の検証は品質の重要な一部。環境整備を優先すべき。賛成。
- **田中（Dev）**: Sprint 002 で環境整備を行い、全 `.tf` ファイルの plan 検証を実施する。賛成。
- **高橋（SM）**: DoD 本文は現行のまま維持し、検証できない場合の例外対応を運用ルールとして追記するのが適切。

**結論**: 完成の定義の本文は変更しない。ただし、`terraform plan` が実行できない環境上の制約がある場合は、障害物ログに記録し、解消後に速やかに検証を実施するという運用ルールをチームで合意した。

## 6. 総括

Sprint 001 は初回スプリントとして計画 46SP を 100% 消化し、スプリントゴールを達成した。チームは依存関係を意識した計画的な開発、明確な役割分担、高品質なテスト整備を実現し、自己管理型チームとしての良い基盤を築けた。

一方で、顧客 Must 要件との認識ギャップ、Terraform 検証環境の未整備、フロントエンド開発の時間集中、障害物ログの未活用という課題も明らかになった。Sprint 002 では 3 つの改善アクションアイテムを実行し、これらの課題に対処する。

**スプリントレトロスペクティブをもって Sprint 001 は終了です。チームの皆さん、お疲れさまでした。**
