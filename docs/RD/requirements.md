## 1. 背景・目的

### 1.1 背景

AWS CloudWatch は強力なモニタリングサービスだが、以下の課題がある。

- UI が複雑で、必要な情報にたどり着くまでのステップが多い
- 小規模アプリケーションの全体状態を一目で把握しにくい
- API エンドポイント単位の運用状況（レイテンシ、エラー率等）を横断的に確認しづらい

個人開発者や小規模チームにとって、Datadog 等の商用ツールはオーバースペックかつコストが高い。

### 1.2 目的

サーバーレス小規模アプリ向けに、必要な運用指標だけを簡単に可視化する Observability Dashboard を構築する。

### 1.3 ゴール

- アプリケーション全体の健康状態を一目で把握できる
- API エンドポイント単位のパフォーマンスを確認できる
- エラーログを迅速に確認・調査できる
- AWS サービスのみで低コスト（月 $1 以下）に運用できる

## 3. 機能要件

| システム | 説明 |

| **Demo Web App** | 監視対象となるシンプルな投稿アプリ |

---

### 3.2 Demo Web App（監視対象アプリ）

#### 3.2.1 概要

シンプルな投稿（CRUD）アプリケーション。Observability Platform の監視対象として動作する。

#### 3.2.2 API エンドポイント

| ID | Method | Path | 説明 | 優先度 |
|----|--------|------|------|--------|
| DA-API-01 | GET | `/posts` | 投稿一覧を取得 | Must |
| DA-API-02 | POST | `/posts` | 新規投稿を作成 | Must |
| DA-API-03 | GET | `/posts/:id` | 特定の投稿を取得 | Must |
| DA-API-04 | DELETE | `/posts/:id` | 特定の投稿を削除 | Must |

#### 3.2.3 データモデル

**Post（投稿）**

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | string (UUID) | 投稿の一意識別子 |
| `title` | string | 投稿タイトル |
| `body` | string | 投稿本文 |
| `createdAt` | string (ISO 8601) | 作成日時 |
| `updatedAt` | string (ISO 8601) | 更新日時 |

#### 3.2.4 機能要件

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| DA-F-01 | 投稿一覧表示 | 全投稿をリスト表示する | Must |
| DA-F-02 | 投稿作成 | タイトルと本文を入力して投稿を作成する | Must |
| DA-F-03 | 投稿詳細表示 | 特定の投稿の詳細を表示する | Must |
| DA-F-04 | 投稿削除 | 特定の投稿を削除する | Must |
| DA-F-05 | 構造化ログ出力 | 全 Lambda 関数で JSON 形式の構造化ログを出力する | Must |

#### 3.2.5 アーキテクチャ

```
User → CloudFront → S3 (Next.js Static Export) → API Gateway → Lambda (Python) → DynamoDB
```

### 3.3 Observability Platform（監視ダッシュボード）

#### 3.3.1 画面一覧

| 画面 ID | 画面名 | 説明 |
|---------|--------|------|
| OP-S-01 | Overview | アプリ全体の健康状態を一目で把握する |
| OP-S-02 | Endpoints | API エンドポイント一覧の運用状況 |
| OP-S-03 | Endpoint Detail | 特定 API の詳細なメトリクス・ログ |
| OP-S-04 | Logs | ログビューア |

#### 3.3.2 OP-S-01: Overview（アプリ全体の状態）

**目的:** アプリの健康状態を一目で把握する

| ID | 表示項目 | 説明 | 優先度 |
|----|---------|------|--------|
| OP-F-01 | 総リクエスト数 | 選択時間範囲内の総リクエスト数 | Must |
| OP-F-02 | エラー率 | 4XX + 5XX エラーの割合 | Must |
| OP-F-03 | 平均レスポンス時間 | 平均レイテンシ | Must |
| OP-F-04 | p95 レイテンシ | 95 パーセンタイルのレスポンス時間 | Must |
| OP-F-05 | 推移グラフ | 選択時間範囲のメトリクス推移を折れ線グラフで表示 | Must |
| OP-F-06 | 時間範囲セレクタ | 1h / 6h / 24h / 7d から選択可能 | Must |

#### 3.3.3 OP-S-02: Endpoints（API 一覧）

**目的:** API ごとの運用状況を一覧で確認する

| ID | 表示項目 | 説明 | 優先度 |
|----|---------|------|--------|
| OP-F-07 | リクエスト数 | エンドポイントごとの呼び出し回数 | Must |
| OP-F-08 | 成功率 | エンドポイントごとの成功率（%） | Must |
| OP-F-09 | 平均レイテンシ | エンドポイントごとの平均レスポンス時間 | Must |
| OP-F-10 | p95 レイテンシ | エンドポイントごとの 95 パーセンタイル | Must |
| OP-F-11 | エラー数 | エンドポイントごとのエラー発生数 | Must |
| OP-F-12 | 時間範囲セレクタ | Overview と同様の時間範囲選択 | Must |

#### 3.3.4 OP-S-03: Endpoint Detail（API 詳細）

**目的:** 特定の API の詳細な動作を確認する

| ID | 表示項目 | 説明 | 優先度 |
|----|---------|------|--------|
| OP-F-13 | レイテンシ推移グラフ | 時間軸での平均・p95 レイテンシ推移 | Must |
| OP-F-14 | エラー数推移グラフ | 時間軸でのエラー数推移 | Must |
| OP-F-15 | ステータスコード内訳 | 2XX / 4XX / 5XX の割合（円グラフまたはバー） | Must |
| OP-F-16 | 最新エラーログ | 直近のエラーログ一覧（最新 20 件） | Must |
| OP-F-17 | 時間範囲セレクタ | Overview と同様の時間範囲選択 | Must |

#### 3.3.5 OP-S-04: Logs（ログビューア）

**目的:** アプリケーションログを確認する

| ID | 表示項目 | 説明 | 優先度 |
|----|---------|------|--------|
| OP-F-18 | ログ一覧 | timestamp / level / endpoint / requestId / message を表示（最新 100 件） | Must |
| OP-F-19 | ログレベルフィルタ | INFO / WARN / ERROR で絞り込み | Must |
| OP-F-20 | エンドポイントフィルタ | 特定のエンドポイントで絞り込み | Should |
| OP-F-21 | 時間範囲セレクタ | Overview と同様の時間範囲選択 | Must |

#### 3.3.6 共通機能

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| OP-F-22 | 手動リロード | ユーザーが明示的にデータを再取得するボタン | Must |
| OP-F-23 | エラー表示 | CloudWatch API からのデータ取得失敗時に、わかりやすいエラーメッセージを UI に表示する | Must |
| OP-F-24 | ローディング表示 | データ取得中のローディングインジケータ | Must |

#### 3.3.7 アーキテクチャ

```
CloudFront → Next.js (Static Export) → API Gateway → Lambda (Python) → AWS SDK (boto3) → CloudWatch API
```

## 5. スコープ外

以下は MVP では対象外とし、将来拡張として検討する。

| 項目 | 理由 |
|------|------|
| 分散トレーシング（X-Ray 連携） | MVP の複雑度を抑えるため |
| OpenTelemetry 完全対応 | MVP の複雑度を抑えるため |
| マルチテナント | 個人・小規模利用のため不要 |
| 高度なアラート・通知 | MVP では可視化のみに集中 |
| ログ全文検索 | CloudWatch Logs Insights の利用で代替 |
| カスタムダッシュボード | 固定レイアウトで MVP を提供 |
| 複数 AWS アカウント対応 | 単一アカウントのみ |
| 複数リージョン対応 | 東京リージョンのみ |
| 認証・認可 | MVP では実装しない |
| 自動リロード / リアルタイム更新 | 手動リロードで対応 |
| バッチデータ収集パイプライン | リアルタイム CloudWatch API 呼び出しで対応 |

## 6. 制約・前提条件

### 6.1 前提条件

| ID | 前提 |
|----|------|
| PRE-01 | AWS アカウントが利用可能であること |
| PRE-02 | AWS CLI が設定済みであること |
| PRE-03 | Node.js >= 18 が利用可能であること |
| PRE-04 | Python 3.12 以上が利用可能であること |
| PRE-05 | Terraform CLI が利用可能であること |

### 6.2 技術的制約

| ID | 制約 | 影響 |
|----|------|------|
| CON-01 | CloudWatch GetMetricData API は 50 TPS 制限 | 手動リロードのため問題なし |
| CON-02 | CloudWatch Logs Insights は同時 30 クエリ制限 | 単一ユーザー利用のため問題なし |
| CON-03 | CloudWatch メトリクスの保持期間は解像度に依存（1分: 15日、5分: 63日、1時間: 455日） | 7 日以上の過去データ表示時は解像度が落ちる |
| CON-04 | Lambda コールドスタート | 初回リクエスト時にレイテンシが増加する可能性 |

### 6.3 技術スタック

| レイヤー | 技術 |
|---------|------|
| **Frontend** | Next.js (Static Export), TypeScript, Tailwind CSS ※ Demo App ・ Observability Platform 共通 |
| **Backend** | AWS Lambda (Python), API Gateway |
| **Data Source** | CloudWatch Metrics, CloudWatch Logs |
| **Database** | DynamoDB（Demo App 用） |
| **Hosting** | CloudFront + S3 |
| **IaC** | Terraform |
| **CI/CD** | GitHub Actions |

## 7. 用語定義

| 用語 | 説明 |
|------|------|
| **Observability** | システムの内部状態を外部から観測可能にする能力。メトリクス、ログ、トレースの 3 本柱で構成される |
| **メトリクス** | システムの定量的な計測値（リクエスト数、レイテンシ、エラー率など） |
| **p95 レイテンシ** | 全リクエストの 95% がこの時間以内に完了するレスポンス時間 |
| **構造化ログ** | JSON 形式でフィールドが定義されたログ。フィルタリングや解析が容易 |
| **コールドスタート** | Lambda 関数の初回実行時に発生する初期化遅延 |
| **MVP** | Minimum Viable Product。最小限の機能で価値を提供するプロダクト |
| **Demo Web App** | Observability Platform の監視対象として構築するサンプルアプリケーション |
| **Observability Platform** | CloudWatch からデータを取得して可視化するダッシュボードアプリケーション |

## 付録

### A. データ取得戦略

**MVP（採用）: リアルタイム取得**

```
Dashboard → API Gateway → Lambda → CloudWatch API → レスポンス
```

- 実装がシンプル
- データパイプライン不要
- 手動リロードのため API コール頻度が低い

**将来拡張: バッチ収集**

```
EventBridge → Lambda → CloudWatch → DynamoDB → Dashboard
```

- パフォーマンス改善が必要な場合に検討

### B. ログ仕様

#### ログフォーマット（JSON 構造化ログ）

```json
{
  "timestamp": "2026-03-15T12:34:56Z",
  "level": "ERROR",
  "service": "demo-api",
  "functionName": "createPost",
  "endpoint": "POST /posts",
  "requestId": "abc-123",
  "message": "Failed to create post",
  "statusCode": 500,
  "durationMs": 842,
  "errorCode": "DDB_WRITE_FAILED"
}
```

#### 必須フィールド（全ログ共通）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `timestamp` | string (ISO 8601) | ログ発生日時 |
| `level` | string | ログレベル（INFO / WARN / ERROR） |
| `service` | string | サービス名 |
| `endpoint` | string | 対象エンドポイント |
| `requestId` | string | リクエスト識別子 |
| `message` | string | ログメッセージ |

#### リクエスト完了ログの追加フィールド

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `statusCode` | number | HTTP ステータスコード |
| `durationMs` | number | 処理時間（ミリ秒） |

#### エラーログの追加フィールド

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `errorCode` | string | エラーコード |
| `errorType` | string | エラー種別 |

#### ログレベル定義

| レベル | 用途 |
|--------|------|
| `INFO` | 通常の操作ログ |
| `WARN` | 警告（処理は継続） |
| `ERROR` | エラー（要対応） |

> **Note:** `DEBUG` は MVP では使用しない（ログノイズ削減のため）。

### C. 取得メトリクス一覧

#### API Gateway メトリクス

| メトリクス | 説明 | 用途 |

| `Latency` | エンドツーエンドのレイテンシ | レイテンシ表示 |

| `Throttles` | スロットリング回数 | 障害検知 |
